# AI token 用量治理规范

本文定义 TOS 项目内 AI/agent 协作的 token 用量红线、审计口径和日常降耗规则。目标不是减少必要验证，而是避免大上下文重复注入、工具循环和无关文件读取造成异常消耗。

## 目标和红线

- 单人一周 `4.3 亿 tokens` 视为异常红线，出现后必须立刻做用量审计。
- 第一阶段降耗目标：在不影响交付质量和必要验证的前提下，把周 token 消耗压到当前的 `20%-40%`。
- 优先审计 `input tokens`。如果 input 占大头，优先排查上下文复用、大文件读取和历史消息重复注入；如果 output 占大头，优先限制输出格式和报告长度。

## 审计维度

每次用量审计至少拉取以下维度：

- 按天 token。
- 按任务/会话 token。
- `input tokens / output tokens`。
- 按模型 token。
- 按工具调用次数、失败重试次数。
- Top 10 最大消耗会话。
- 高消耗会话中的主要上下文来源：需求文档、Excel/PDF/log、源码目录、历史对话、命令输出。

## Agent 工作规则

- 大任务开始前先形成结构化摘要：业务目标、输入材料、相关文件清单、验收标准、明确不做的范围。
- 业务需求原始材料先摘要再进入开发 agent；开发 agent 只拿必要字段、相关源码片段和验收标准，不接收完整历史。
- 不默认读取完整仓库、完整历史对话、整份 Excel、整批 PDF 或完整日志。先定位目录、文件类型、关键字段和失败片段，再读取必要范围。
- 搜索和读取默认排除无关大文件和产物目录，例如 `node_modules`、`dist`、`build`、`*.min.js`、`package-lock.json`、`recovered-frontend`、`archive`、打包产物和缓存目录。
- 多 agent 或多线程并行时，不能把完整上下文复制给每个 agent。每个 agent 只接收自己的任务摘要、相关文件和验收标准。
- 输出默认保持短结论：改了什么、验证了什么、风险是什么。除非明确要求，不输出长篇方案解释、完整日志或大段代码复述。

## 异常处置

出现以下情况时，先暂停继续扩展上下文，改做用量审计和任务收敛：

- 同一文件、目录或命令被反复读取，但没有产生新的定位信息。
- 命令失败后进入重复重试，没有先分析失败原因或缩小验证范围。
- input tokens 明显高于 output tokens，且会话中包含大文件、历史对话或完整命令输出。
- 并行 agent 数量增加后，总 token 近似成倍增长。
- Top 消耗会话集中在“需求对接 + 半自动化工具开发”这类本应局部定位的任务。

处置顺序：

1. 记录当前任务、会话、模型、工具调用次数和 input/output token。
2. 抽取当前上下文摘要，删除无关历史和完整大文件内容。
3. 将后续任务拆成最小可验证单元，每个单元只保留相关文件和验收标准。
4. 对输出加限制，例如“只输出 diff 摘要、验证结果和剩余风险”。
5. 复盘 Top 10 高消耗会话，补充新的排除规则或任务模板。
